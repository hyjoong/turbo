> # 들어가며

최근에 코드 리뷰를 하면서 생각이 많아지면서 웹 개발기간 3년 동안 어떻게 코드 리뷰를 해왔고, 코드 리뷰를 통해 어떻게 성장했는지, 어떤 코드 리뷰 방식을 선호하는지를 돌아보고자 한다.

---

> ## 1. 취업 준비 (2021.02 ~ 2021.09)

대학교 4학년 2학기가 끝나자마자 웹 프론트엔드 부트캠프를 신청하여 프론트엔드 개발 공부를 시작했었다.(~~수업 듣느라 졸업식도 안 갔다~~) 부트캠프에는 30명의 수강생과 5명의 멘토가 있었는데, 코드리뷰에 적극성은 사람마다 달랐다. 간단한 코멘트만 남기는 사람도 있었고, 적극적으로 리뷰에 참여하는 사람도 있었지만 나는 적극적인 편에 속했던 것 같다.
첫 코드 리뷰는 팀원 4명씩 구성된 팀끼리 리뷰를 했었고, 아래는 내가 처음으로 코드리뷰 했던 모습이다.
![](https://velog.velcdn.com/images/hyunjoong/post/757ae2d3-558c-4dbe-a4d8-003af0e42f32/image.png)

왜 이렇게 모든 리뷰마다 뒤에 이모티콘을 붙였는지 기억이 안 나지만, 이 방식도 나쁘지 않은 방법 같다🙋‍♂️ 멘토 분들 리뷰 중에서는 특히 신영 님의 리뷰가 가장 기억이 남는다. 신영님은 칭찬을 먼저한 후 개선점을 제안하는 방식으로 리뷰를 진행하셨고, 이때 신영님에게 코드리뷰를 많이 배울 수 있었다.

> ## 2. 첫 번째 회사 (2021.10 ~ 2022.03)

2021년 10월, 첫 직장에서 6개월 인턴으로 일을 했다. 당시 회사에는 개발자가 CTO와 나뿐이었고, 프론트엔드 개발을 혼자서 담당하다 보니 코드 리뷰를 받을 기회가 적었다. 인턴 기간이 끝난 후 정규직 전환을 제안받았지만, 코드리뷰와 고민을 나눌 개발자가 없다는 것이 가장 큰 이유로 거절하게 됐었다.

이때는 회사에서 혼자 개발을 담당하면서 성장할 환경이 부족했지만 외부 교육 과정의 코드리뷰를 보거나 프로젝트를 하면서 성장을 했었다.

[코드숨](https://github.com/CodeSoom), [우아한 테크코스](https://github.com/woowacourse), [우테코 팀프로젝트](https://github.com/woowacourse-teams)에서 수강생들이 작성한 코드의 PR과 멘토분들이 작성한 리뷰를 보며 학습했었다. 특히 우테코 과정에서 Jbee님은 기술적으로 리뷰를 잘 남겨주셔서 Jbee님이 리뷰한 PR위주로 보기도 했었다. 아래는 Jbee님이 남긴 리뷰의 일부다.

- [Jbee님의 리뷰1](https://github.com/woowacourse/react-shopping-cart/pull/9#discussion_r632929757)
- [Jbee님의 리뷰2](https://github.com/woowacourse/react-shopping-cart/pull/16#pullrequestreview-660342744)

그리고 Jbee님의 [코드 리뷰의 목적은 성장이어야 한다](https://www.jbee.io/articles/essay/%EC%BD%94%EB%93%9C%20%EB%A6%AC%EB%B7%B0%EC%9D%98%20%EB%AA%A9%EC%A0%81%EC%9D%80%20%EC%84%B1%EC%9E%A5%EC%9D%B4%EC%96%B4%EC%95%BC%20%ED%95%9C%EB%8B%A4) 글을 통해 코드리뷰 목적을 더 이해할 수 있었다.

> ## 3. 두 번째 회사 (2022.05 ~ 2023.11)

두 번째 회사에서는 프론트엔드 개발자가 10명이 있었다. 나는 내가 담당하지 않은 프로젝트의 PR 코드도 자주 보며 의견을 나누었고, 혼자 프로젝트를 담당했을 때는 다른 개발자에게 코드 리뷰를 요청하며 적극적으로 코드리뷰를 했었다.

가장 기억에 남는 리뷰는 한 팀원이 로그인하지 않은 사용자가 특정 페이지에 접근하지 못하도록 클라이언트 단에서 라우트 가드를 통해 리다이렉트 로직을 처리한 코드에 대한 리뷰였다. 이 리뷰가 특히 기억에 남는 이유는 내가 다른 의견을 제시하고 설득하기 위해 자료를 조사하고 다양한 비유를 사용했기 때문이다.

나는 Next.js의 middleware을 활용하여 라우트 가드를 설정하는 것을 제안하고 서버사이드 비용이 들 수 있으니 미들웨어가 실행될 경로(matcher)를 잘 설정해야 한다고 강조했다. 내 리뷰를 받은 팀원이 비용 문제로 사용을 꺼려했을 때, 나는 회사에서 사용중인 Vercel Enterprise 플랜이 한 달에 100만 번의 미들웨어 실행을 기본적으로 제공하고 이 제공량을 초과해도 100만 번당 0.65달러밖에 추가 비용이 들지 않는다고 설명했다.
이것을 마치 한 달에 데이터 100GB를 제공하는 요금제를 사용하면서 데이터 비용이 아까워 와이파이로만 인터넷을 사용하는 것에 비유를 했었다.

물론, 내가 제안한 방식이 무조건 정답은 아니고 각자의 장단점이 있겠지만, 예시까지 들어 설명해서 특히 기억에 남는다.

---

하지만 개발자들마다 리뷰를 받아들이는 자세와 스타일의 차이로 어려움을 겪기도 했었다. 리뷰를 주고받을 때 내가 감정적으로 반응했던 적도 있었고, 코드리뷰의 방식이 서로 달라서 협업이 어려웠던 적도 있었다.

코드 리뷰를 진행하면서 의도와 다르게 상대방이 받아들일 수 있다는 것을 깨달았고, 이에 따라 논쟁이 될 수 있는 리뷰는 먼저 구두로 논의한 후 리뷰를 남기는 방식을 사용하여 서로 감정이 상하지 않도록 노력했었다.

---

업무 외적으로는 [next-step](https://github.com/next-step) 교육과정의 코드리뷰를 통해 많이 배웠다. 유명 기업(네이버, 배민, 당근)의 멘토들이 작성한 리뷰를 보며 많이 배웠고 특히 당근의 정현수님의 리뷰를 보고 많이 배웠다.

- [정현수님의 코드리뷰](https://github.com/next-step/react-payments/pulls?q=assignee%3Ajunghyeonsu+is%3Aclosed)를 참고하면 좋은 코드리뷰 예시를 볼 수 있다

![review](https://velog.velcdn.com/images/hyunjoong/post/992b52d6-33d2-4f14-886b-5d9193dcefcb/image.png)

위의 사진은 정현수님이 작성한 코드리뷰의 일부다. 코드 리뷰에서 제안하는 방식을 사용하여 리뷰어의 생각을 강요하지 않고, 제안된 방식이 다를 경우에도 편하게 논의할 수 있도록 해주는 방식이다. 예를 들어,

> digits3, digits4 보다 digits.thridSection, digits.fourthSection 방식이 더 좋을 것 같습니다.

> digits.thridSection, digits.fourthSection 와 같이 명시적으로 작성할 수 있겠네요.

이렇게 리뷰를 작성할 수도 있었겠지만, 이런 리뷰는 어느정도 리뷰어의 생각을 강요하는 부분이 있다. 팀원 간의 컨벤션을 맞추어야 하는 부분에서는 팀원에게 선택권을 주는 것이 아니라 어느 정도 강요하는 방식이 더 좋을 수도 있지만 그게 아니라면, 상대방에게 부담감을 줄 수도 있다. 만약 이러한 리뷰를 받고 내가 digits3, digits4 방식이 더 좋다고 생각해서 내 생각을 말하게 된다면 리뷰어가 내가 코드 리뷰를 받아들이지 않는 부정적인 사람으로 생각하게 될까 봐 내 의견을 표출하지 못 할 수도 있다.(개인적인 생각 입니다)

코드 리뷰를 하면서 많이 고민했던 부분은, 상대방이 제안한 코드가 내가 생각하는 방식과 다를 때, 이를 논의하고 설득하는 시간 비용과 언쟁이 일어날 가능성을 고려해, 그냥 받아들이는 것이 더 효율적일 수도 있다는 생각이 들어서 반영했던 경우도 종종있었다. 하지만 정현수님이 작성한 리뷰처럼 상대방이 부담을 느끼지 않게 리뷰를 남긴다면, 상대방도 더 편하게 리뷰에 대해 논의할 수 있을 것 같다.

> # 결론

코드 리뷰에 관심이 많고 적극적으로 참여하는 것이 나의 장점이라고 생각했었다. 그러나 최근 코드 리뷰를 통해 트러블이 생기는 경우가 많아지면서, 내 전달 방식이 잘못된 것은 아닌지 고민하게 되었다. 내가 코드 리뷰의 목적을 충분히 이해하지 못하고 상대방에게 부담을 주고 있지는 않나 되돌아보게 되었다. 개발 경험동안 코드 리뷰 문화를 경험하면서 다음과 같은 성장을 이루게 되었다.

## 코드 리뷰를 통한 성장

### 1. 코드 품질 향상

코드 리뷰를 통해 팀원과 코드 스타일이나 컨벤션을 맞추는 데 도움이 되어, 팀원이 퇴사하거나 프로젝트 담당자가 바뀔 때도 코드 작성에 어려움이 없었고, 새로운 팀원이 빠르게 프로젝트에 적응할 수 있었다.

### 2. 커뮤니케이션 능력

코드 리뷰를 통해 다른 개발자들과 소통 능력이 향상됐다. 서로 다른 의견을 제안하고 설득하며 타협하는 과정을 통해 협업 능력을 기를 수 있었다. 특히, 감정이 상할 수 있는 리뷰를 보더라도 상대방의 리뷰 스타일이 다를 수 있다는 점을 이해하고, 감정적으로 받아들이지 않으려 노력했다.

### 3. 문제 해결 능력

문제를 다양한 시각에서 접근하고 해결하는 능력이 향상되었다.

---

이러한 성장의 기반에는 내가 가진 몇 가지의 기준점이 있다.

## 코드 리뷰의 기준점

### 1. 긍정적인 피드백

항상 긍정적인 피드백을 먼저 제공하고, 그 후에 개선점에 대해 제안하는 방식으로 접근해서 팀원이 부담 없이 리뷰를 받아들이게 한다.

### 2. 구체적인 예시 제공

가능하면 예시 코드와 참고 자료를 함께 제공하면서 제안한 의도를 더 명확하게 전달한다.

---

코드 리뷰에 적극적이지 않고 단순히 승인만 눌렀다면 서로 트러블이 없이 좋게 지나갈 수 있는 경우가 많다. 그러나 코드 리뷰는 단순히 잘못된 점을 지적하는 것이 아니라 함께 성장하는 과정이라고 생각한다. 코드 리뷰는 여전히 어려운 부분이 많지만 코드 리뷰를 통해 성장하고 배울 수 있다는 점에서 그 중요성은 매우 크다고 느낀다. 앞으로 더 나은 방식으로 코드 리뷰를 진행하기위해 고민을 많이 해야겠다.
